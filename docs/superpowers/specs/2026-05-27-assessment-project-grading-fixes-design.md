# 练习与项目判分修复设计

> 日期：2026-05-27  
> 状态：已于 2026-05-27 书面确认

## 目标

修复目前会破坏学习闭环的两类问题：

1. 练习题出现正确答案被判错、错误代码被判对、答案与解析不一致。
2. 项目实战任务因为随机输出和交互输入不足而无法稳定运行或提交通过。

本次修复以“正确行为可稳定通过，错误行为不能误过，参考答案不暴露给浏览器”为验收标准。Dashboard、学习进度 UI 和通用视觉打磨不属于本次范围。

## 当前根因

### 练习题

- 判断题种子答案为 `A` / `B`，前端提交为 `true` / `false`，后端判题未统一映射。
- 个别填空题的题干、可接受答案和 UI 限制互相冲突，例如允许 `python3` 却只接受 `python`，以及正确答案为空但空输入无法提交。
- `code_fix` 和 `code_completion` 题缺少可运行的初始代码；现有判题只比较输出，导致 `pass` 可被误判通过。
- `programming` 题没有测试用例；现有回退策略把任意可运行代码判为正确。

### 项目实战

- 项目任务均无测试用例，提交逻辑通过分别运行用户代码和参考答案、比较完整 stdout 判分。
- 随机项目的参考答案自身输出不确定，无法用 stdout 全等比较。
- 菜单及循环项目需要多轮标准输入，按代码文本统计 `input()` 次数无法覆盖运行路径。
- 项目详情响应含 `answer_code`，浏览器可读取参考答案。
- 页面标准输入仅为单行输入框，不能便捷表达交互项目所需的多行输入。

## 判分契约

### 客观题

- `choice` 继续按选项标签判分。
- `judge` 的公共提交契约为布尔语义值：前端可继续提交 `true` / `false`；后端将题库中现有 `A` / `B` 以及中文或英文布尔表达统一归一化后比较。
- `fill_blank` 支持每题声明一个或多个可接受文本答案，并在比较前移除首尾空白。只有语义确实等价且题面明确允许时才设置多个答案，例如 `python` 与 `python3`。
- 需要空字符串作答案的题目改写为可提交的非空答案形式，不依赖空输入绕过 UI 限制。

### 代码练习

- `code_fix`、`code_completion` 与 `programming` 均提交一段完整、可执行的用户代码。
- `code_fix` 提供包含缺陷的 `starter_code`；用户编辑完整程序后提交。
- `code_completion` 提供带待补位置的 `starter_code`，前端将其作为编辑器起始内容；用户提交补全后的完整程序，不再将答案片段与缺失上下文拼接执行。
- `programming` 继续从空白或题目给定模板开始，但必须具有至少一个可验证的测试用例。
- 每个代码题通过一组测试用例运行用户代码。每条测试用例至少包含 `stdin` 和 `expected_output`；全部通过才为正确。
- 不再使用“没有测试用例则程序只要退出码为 0 就通过”的策略。数据缺少测试用例时应判为配置错误，而不是放行用户答案。

## 项目任务判分

- 每个 `ProjectTask` 配置确定性的测试用例；提交时运行用户代码对测试用例逐项验证，不运行并比较对外不可见的参考答案输出。
- 每条测试用例包含多行 `stdin` 和可比较的 `expected_output`。交互提示语属于题目体验的一部分时可以包含在期望输出中。
- 猜数字项目需将随机行为转为可测行为。种子任务可以通过固定随机种子或把可预测输入/输出写入任务模板，使自动测试不会依赖随机碰运气。
- 待办事项项目使用包含退出操作的多行输入场景，验证添加、查看、完成、删除等目标行为后程序能够结束。
- 参考答案仅保留在后端数据或维护脚本中，用于内容维护和验证测试用例；不得出现在项目详情或任务列表 API 响应中。

## API 与前端边界

### 练习 API

- 获取题目列表和详情时，不返回 `answer`、`reference_code` 或测试预期输出。
- 详情可返回安全的 `starter_code`，供用户编辑。
- 提交响应在判题完成后可返回解释文字和本次执行输出，不返回标准答案源代码。

### 项目 API

- 获取项目详情和任务列表时，返回任务标题、说明、提示与 `starter_code`，省略 `answer_code` 和私有测试用例。
- 提交响应只包含通过状态及用户运行过程中的可展示输出或错误，不包含参考答案。

### 页面交互

- 练习页继续提交完整编辑器内容作为代码答案。
- 项目页标准输入控件改为多行文本区域，便于用户手动运行需要多轮输入的程序。
- 自动判分使用后端定义的测试输入，不依赖用户在“运行”面板中填写的临时 stdin。

## 数据更新策略

- 源数据继续由 `backend/app/seed/data/exercises.py` 与 `backend/app/seed/data/projects.py` 管理。
- 为已存在的本地 SQLite 数据库提供可重复执行的数据同步方式，只更新课程内容中的题目和任务配置字段，不删除用户、进度、尝试记录或代码提交记录。
- 更新内容至少包括：
  - 判断题答案契约可被新判题器接受。
  - 冲突填空题的题干与可接受答案。
  - 代码练习的 `starter_code` 与 `test_cases`。
  - 项目任务的可测模板和 `test_cases`。
- 新建数据库重新播种后，以及现有数据库执行同步后，均应得到一致的可判分内容。

## 错误处理

- 代码练习或项目任务如果缺少测试用例，提交请求返回明确的内容配置错误，不将其视为用户答错。
- 用户代码运行超时、异常退出或输出不匹配时，判为未通过，并返回可读的执行反馈。
- 对于多个测试场景，反馈至少说明是否存在未通过的场景；不向用户泄露隐藏测试输入或标准答案。

## 测试与验收

### 后端自动测试

- 判断题：前端使用的 `true` / `false` 提交路径可以正确通过或判错。
- 填空题：题面声明的等价答案可通过；不再存在无法由 UI 提交的正确答案。
- 纠错题：参考修复代码可通过，`pass` 等不完成任务的代码不能通过。
- 补全题：补全后的完整参考代码可通过，空实现不能通过。
- 编程题：参考实现逐题可通过；`pass` 不能通过任何要求输出或处理数据的题目。
- 项目任务：10 个任务的参考实现均通过各自测试场景；明显无效实现不能通过。
- API 输出：练习与项目公共读取端点均不包含答案、参考代码或隐藏测试用例。

### 前端验证

- 练习页能够展示并提交完整的修复/补全模板。
- 项目页允许输入多行 stdin 手动运行。
- TypeScript 类型检查通过，包括现存 `RecentActivity` 类型错误的修复。

### 回归验证

- 后端 Python 语法检查通过。
- 后端自动测试通过。
- 前端 `tsc --noEmit` 通过。
- 使用保留数据的同步流程后，现有本地数据库中的课程、题目、项目数量保持不变，用户数据不被清空。

## 不在本次范围

- Dashboard 最近动态与骨架屏。
- 课程锁定逻辑和完整学习进度展示。
- 全局错误边界与移动端整体适配。
- 代码执行沙箱的生产部署加固与接口限流；这些仍是上线前必要工作，但与本次判分修复分开处理。
